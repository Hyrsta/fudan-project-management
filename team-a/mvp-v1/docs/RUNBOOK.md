# Team A MVP Runbook

## Service overview

Team A Final Product v1 is a local-first FastAPI application that creates a
structured news intelligence report from a topic query. It is built for course
demos, so the system prefers graceful degradation over hard failure.

## Intended audience

This runbook is for developers and operators who need to:

- start the app locally
- verify that it is healthy
- understand the main runtime paths
- troubleshoot common failures before a demo

## Dependencies

### Runtime

- Python 3
- Node.js and npm for rebuilding the React frontend
- Network access for live Google News RSS retrieval
- Optional OpenAI-compatible API access for model-written sections
- Trusted RSS feeds plus Google News RSS fallback for live retrieval

### Python packages

Installed from:

- `requirements.txt`
- `requirements-dev.txt`

## Startup

### Recommended quick start

From `team-a/mvp-v1/`:

```bash
./run_local.sh
```

What this does:

1. creates `.venv` if missing
2. activates the virtual environment
3. installs `requirements-dev.txt`
4. runs `python -m news_brief_mvp`

### Manual startup

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements-dev.txt
python -m news_brief_mvp
```

### Manual uvicorn startup

```bash
source .venv/bin/activate
uvicorn news_brief_mvp.main:app --host 127.0.0.1 --port 8000 --reload
```

## Access control

RBAC is enabled by default for recent briefs, saved reports, exports, handoff
JSON, and brief generation. The React shell and health check stay public so the
local demo can load before an access key is selected.

Default local demo keys:

- `viewer-local-token`: read saved briefs and HTML/Markdown exports
- `analyst-local-token`: create briefs and read HTML/Markdown exports
- `admin-local-token`: create briefs, read exports, and open Team B handoff JSON

For command-line checks, pass the key with `X-API-Key`:

```bash
curl -H "X-API-Key: admin-local-token" http://127.0.0.1:8000/briefs/<brief_id>/handoff
```

The browser login page validates the selected demo account or custom key against
`GET /api/briefs/recent`, then stores the accepted key in the local auth cookie.

For custom tokens, set:

```bash
export NEWS_BRIEF_RBAC_TOKENS="viewer:read-token;analyst:write-token;admin:admin-token"
```

Disable RBAC only for isolated local troubleshooting:

```bash
export NEWS_BRIEF_RBAC_ENABLED=false
```

## Request flow

### High-level flow

1. user enters a topic
2. app receives `POST /api/briefs`
3. service attempts trusted RSS and Google News RSS retrieval unless mode is fallback
4. articles are scored by source weight, freshness, and topic match
5. duplicate or low-value candidates are filtered out
6. sections are generated by:
   - LLM first
   - heuristic builder second
   - precomputed fallback brief last, when relevant
7. artifacts are written locally
8. result is rendered in the UI or returned as JSON

## Main components

### `news_brief_mvp/main.py`

- FastAPI app factory
- route registration
- React app shell, HTML report, and JSON response handling
- route-level RBAC dependency wiring
- `/static` frontend asset mounting

### `news_brief_mvp/auth.py`

- RBAC role and permission map
- API key header/cookie authentication
- environment-driven token configuration

### `news_brief_mvp/templates/`

- Jinja2 page and partial markup
- server-rendered saved report views and partials

### `news_brief_mvp/static/`

- `css/app.css` for page layout and component styling
- `js/app.js` for browser actions, HTMX request headers, access role UI, and local preferences
- `react/` for the Vite-built React app served from `/`

### `frontend/`

- React + TypeScript source for the primary browser app
- `npm run build` compiles `src/main.tsx` into `news_brief_mvp/static/react/`

### `news_brief_mvp/service.py`

- orchestration layer
- live/fallback mode handling
- section generation selection
- artifact persistence coordination

### `news_brief_mvp/live_retriever.py`

- fetches trusted direct RSS results and Google News RSS fallback results
- maps source names
- filters low-trust sources through the source registry

### `news_brief_mvp/ranking.py`

- computes article scores
- deduplicates similar stories
- returns top-ranked selected sources

### `news_brief_mvp/llm.py`

- calls OpenAI-compatible chat completions
- loads prompt template
- expects valid JSON output

### `news_brief_mvp/local_sections.py`

- creates heuristic sections from selected articles
- keeps the app usable when model generation fails

### `news_brief_mvp/storage.py`

- writes brief artifacts to disk
- reloads prior briefs
- lists recent briefs for the authenticated React workspace

## Runtime modes

### Retrieval mode

- `auto`
  - tries live retrieval first
  - falls back to dataset if live results are too weak
- `live`
  - requires sufficient live retrieval success
- `fallback`
  - skips live retrieval and uses the curated dataset

### Section generation mode

- `llm`
  - sections came from the OpenAI-compatible model
- `heuristic`
  - sections came from the local fallback builder
- `precomputed`
  - sections came from the curated dataset payload

## Artifacts

Artifacts are written to:

```text
artifacts/<brief_id>/
```

Files:

- `brief.json` - full structured brief response
- `handoff.json` - Team B handoff payload
- `brief.html` - standalone HTML export
- `brief.md` - Markdown report export

The artifact root also stores:

- `manifest.json` - recent-report index for the authenticated workspace

## Common demo workflow

1. start the app with `./run_local.sh`
2. open `http://127.0.0.1:8000`
3. sign in with a demo role or custom API key
4. submit a topic in `auto` mode
5. if live retrieval is weak, allow fallback mode to complete the demo
6. use recent briefs to reopen prior successful runs
7. show the HTML export, Markdown export, or Team B handoff JSON if needed

## Troubleshooting

### Issue: homepage does not open

**Symptoms**
- browser cannot connect to `127.0.0.1:8000`

**Checks**
- confirm the process is still running
- confirm the terminal did not exit with an import error
- run:

```bash
curl http://127.0.0.1:8000/health
```

**Resolution**
- restart with `./run_local.sh`
- if port 8000 is occupied, stop the old process or run uvicorn on another port

### Issue: model-written sections are not appearing

**Symptoms**
- UI shows `section_generation_mode` as `heuristic` or `precomputed`

**Checks**
- confirm `OPENAI_API_KEY` is set
- confirm `OPENAI_BASE_URL` and `OPENAI_MODEL` are correct
- check whether the API endpoint is reachable

**Resolution**
- set a valid API key and rerun
- if the demo must continue, proceed with heuristic output and explain the fallback behavior honestly

### Issue: live retrieval fails or returns too little coverage

**Symptoms**
- warnings mention incomplete live results
- mode shifts to `fallback`

**Checks**
- confirm network access
- confirm Google News RSS is reachable from the current machine

**Resolution**
- use `auto` mode and let the dataset fallback complete the demo
- use `fallback` mode directly if you want a predictable class demo

### Issue: tests fail with import errors

**Symptoms**
- `ModuleNotFoundError: news_brief_mvp`

**Checks**
- confirm you are running from `team-a/mvp-v1/`
- confirm `pytest.ini` exists

**Resolution**
- run:

```bash
pytest -q
```

- if needed, activate the virtual environment first:

```bash
source .venv/bin/activate
pytest -q
```

## Current limitations

- persona lenses are local configuration, not database-backed user profiles
- live retrieval currently depends on Google News RSS only
- storage is local filesystem based, not database backed
- fallback dataset is narrow and demo-oriented
- this is still an MVP, not a production multi-agent platform

## Demo guidance

For the course demo, emphasize that the MVP already proves five things:

- topic-to-brief workflow
- source ranking and deduplication
- graceful degradation under failure
- export and handoff artifact generation
- a clean local-first demo path
